# Noloong

Noloong 是一个以 agent 会话为核心的本地优先交互系统。它把桌面 GUI、Telegram、微信等客户端都视为同一套 agent runtime 的交互入口。

## Language

**交互客户端**:
面向用户的入口，用来向 agent 会话发送消息并展示运行过程与最终回复。一个 agent 会话可以被不同交互客户端观察或驱动。
_Avoid_: 前端、壳、聊天页

**主交互客户端**:
Noloong 默认面向用户的交互客户端，打开后即可创建或继续 agent 会话。它默认拥有本地内嵌 runtime，但也可以作为高级模式连接外部 runtime。
_Avoid_: 外部控制面、纯配置应用

**配置入口**:
主交互客户端中用于编辑 profile、插件、MCP、存储和运行策略的辅助区域。它服务于交互体验，不是 app 的默认落点。
_Avoid_: 主界面、profile 应用

**Chat 画布**:
主交互客户端中承载会话列表、transcript、运行活动和输入区的沉浸式区域。它强调对话和运行过程，不采用配置入口的表单式布局。
_Avoid_: 设置页、配置表单

**Noloong-native 高保真复刻**:
在 Noloong 自身品牌、窗口结构和交互模型内复刻参考产品的体验细节，尤其是布局节奏、输入区手感和流式输出动效。它不是复制参考产品的身份、文案或品牌资产。
_Avoid_: pixel-perfect 照搬、外观克隆

**agent 会话**:
一个持续存在的 agent 工作上下文，包含消息历史、运行状态、工具请求和可被交互客户端订阅的展示事件。
_Avoid_: chat、thread

**当前会话**:
交互客户端中当前被用户查看和输入的 agent 会话。其它 agent 会话可以继续运行，并在会话列表中显示运行状态。
_Avoid_: 当前聊天、活动窗口

**会话工作目录**:
agent 会话执行工具和解释相对路径时使用的本地目录。交互客户端只能展示或设置真实运行上下文；切换工作目录不应改变已经运行中的 run。
_Avoid_: 装饰性路径、仅 UI 目录

**会话列表**:
交互客户端展示多个 agent 会话的导航区域。它用于切换、创建和观察会话状态，不保存独立于 agent 会话的聊天历史。
_Avoid_: 本地聊天列表、app-only transcript

**会话标题**:
agent 会话用于在会话列表和标题栏中显示的短名称。默认可由第一条用户消息本地生成，后续可由用户手动重命名，并作为 agent 会话 metadata 持久化。
_Avoid_: GUI 私有标题缓存、额外模型标题请求

**输入区**:
当前会话底部的消息输入和运行控制区域。它只能展示真实可用或明确禁用的能力，不显示伪造的模型、工具、令牌或记忆状态。
_Avoid_: composer、假控制栏

**附件输入**:
输入区把用户选择或拖拽的本地文件作为消息内容发送给 agent 会话。附件必须落到真实的消息 media block；不能为了视觉效果显示尚未接入运行时的假文件、假缩略图或假上传状态。
_Avoid_: 装饰性附件、假媒体预览

**运行活动**:
当前会话中工具执行、审批请求、暂停、失败等非文本回复的用户可见状态。运行活动属于 transcript 的上下文，但默认应低干扰、可折叠。
_Avoid_: 控制台日志、裸 JSON、后台噪声

**停止运行**:
交互客户端对当前会话正在执行的 run 发出的中止请求。它只停止当前 run，不拒绝某个审批，也不删除或关闭 agent 会话。
_Avoid_: 关闭会话、拒绝审批、取消消息

**transcript**:
当前会话中用户消息和 assistant 最终消息形成的稳定对话记录。历史 transcript 以 agent 会话状态为准；运行中的流式回复和运行活动由展示事件补充。
_Avoid_: event log、GUI 私有历史

**展示事件**:
agent 会话面向交互客户端发出的用户可见事件，用于表达运行开始、流式回复、思考展示、工具状态、审批请求、失败和完成。
_Avoid_: 原始事件、日志

**interaction 协议**:
交互客户端与 agent runtime 通信的 JSON-RPC 协议。桌面 app 即使使用内嵌 runtime，也通过本地 loopback interaction 协议通信，而不是直接持有 registry。
_Avoid_: GUI 专属 registry 快捷路径、私有内存协议

**流式回复**:
通过展示事件逐步呈现的 assistant 文本。交互客户端只消费展示事件；如果展示事件表达力不足，应扩展展示事件，而不是读取原始事件。
_Avoid_: provider stream、raw event stream

**流式尾迹**:
流式回复中新到达文本片段的短暂视觉状态；片段先以较低透明度进入布局，再在很短时间内过渡为稳定亮度。它不是逐字弹跳动画，也不是模拟 provider 以外的内容。
_Avoid_: 打字机动画、逐字弹跳

**思考展示**:
交互客户端对 assistant reasoning 过程的用户可见呈现，由展示事件驱动。有 reasoning summary 时始终优先展示 summary；只有存在且允许展示的 reasoning 原文才展示原文；思考结束后应折叠为耗时摘要。
_Avoid_: 永久展开的推理日志、伪造 memory

## Example Dialogue

开发者：这次要给桌面端加 Chat，是不是直接做一个新的 chat runtime？

领域专家：不要叫 chat runtime，也不要绕过 agent 会话。桌面端是一个交互客户端，它订阅 agent 会话的展示事件并发送用户消息。

开发者：那桌面端默认只是连一个外部服务吗？

领域专家：不是。作为主交互客户端，它默认内嵌 runtime，让用户打开 app 就能使用；连接外部 runtime 是高级模式。

开发者：没有配置时是不是直接打开配置页？

领域专家：不要让配置入口成为主体验。主交互客户端默认落在 Chat；没有可用配置时，在 Chat 中引导用户进入配置入口。

开发者：Chat 也复用设置页的表单布局吗？

领域专家：不要。Chat 画布是主交互客户端的沉浸式区域，配置入口才使用结构化表单布局。

开发者：那参考视频里的流式文字怎么命名？

领域专家：这是 Noloong-native 高保真复刻的一部分。我们复刻流式输出的手感和动画，但仍使用 Noloong 的品牌与展示事件模型。

开发者：GUI 要不要订阅原始事件来做更丰富的动画？

领域专家：不要。GUI 作为交互客户端只消费展示事件；如果流式回复或工具状态缺少语义，就扩展展示事件。

开发者：内嵌 runtime 时 GUI 能不能直接拿 registry 调接口？

领域专家：不要。桌面 app 也是交互客户端，应该通过 interaction 协议连接 runtime；内嵌 runtime 只是部署方式。

开发者：那每个字要不要单独弹出来？

领域专家：不要做逐字弹跳。流式尾迹表达的是新到达文本逐渐稳定的感觉，既保留真实 delta 节奏，也避免视觉抖动。

开发者：有 reasoning 原文时要不要完全隐藏？

领域专家：不需要完全隐藏，但思考展示优先使用 reasoning summary。只有存在且允许展示的 reasoning 原文才展示原文，并在结束后折叠成耗时摘要。

开发者：如果用户切换会话，正在运行的会话会暂停吗？

领域专家：不会。当前会话只是交互客户端的视图选择；其它 agent 会话继续运行，并通过会话列表展示状态。

开发者：会话列表里的标题是不是 GUI 自己保存一份？

领域专家：不要。会话标题属于 agent 会话 metadata；GUI 可以用第一条用户消息生成默认标题，但不能维护第二套标题来源。

开发者：Chat 顶部显示的目录可以只是一个方便用户看的路径吗？

领域专家：不行。会话工作目录是 agent 会话真实运行上下文，不能只是装饰性 UI 文案。

开发者：为了贴近参考视频，可以先放一些假工具数量和假记忆状态吗？

领域专家：不可以。输入区可以高保真复刻布局和手感，但状态必须来自真实能力；没有真实语义的控件要禁用或隐藏。

开发者：附件能不能先做成漂亮的占位 chip？

领域专家：不要。附件输入必须生成真实的消息 media block；当前无法发送的附件能力就先隐藏或禁用。

开发者：工具输出应该直接作为普通 assistant 消息插进 transcript 吗？

领域专家：不要。工具和审批是运行活动，应该就地呈现但低干扰；用户需要时可以展开细节。

开发者：输入区的停止按钮是不是等同于拒绝当前审批？

领域专家：不是。停止运行只是中止当前 run；审批的同意或拒绝仍由审批卡片表达。

开发者：切换会话或重启 app 后，之前的流式增量从哪里恢复？

领域专家：稳定 transcript 来自 agent 会话状态中的最终消息；正在运行的部分由展示事件继续补充。不要让 GUI 本地缓存成为第二套事实来源。
